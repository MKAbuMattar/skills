Kubernetes
Fundamentals

From Docker containers to a self-managing fleet

Pragmatic K8s for backend & frontend devs · Press ? for help
Welcome. This is part two of the container journey. Docker showed you how to package an app. This deck shows you how to operate a hundred of them.

WHY WE'RE HERE

Goals for today

  • Why we moved — from VMs and bare containers to Kubernetes
  • The mental model — pods, deployments, services, control plane
  • The win — self-healing and elastic scaling
  • Our setup — <Cloud Provider> compute + managed Kubernetes + container registry Enterprise
  • Your cluster — what's actually running, right now
Five takeaways. The last one is special — we'll look at our actual cluster, with real numbers, at the end.

AUDIENCE

"How do I run 50 containers?"

If that question has come up — Kubernetes is the answer.

Show of hands — who's run docker compose in production? Good. That's where this question first hits. K8s is what you reach for when one machine isn't enough.

PART 1

Containers shipped. Now what?

Containers solved packaging. Now we have a different problem: keeping them alive, balanced, scaled, and healthy without a human in the loop.

THE PROBLEM

Docker packaged the app.
Who packages the
operations?

When containers were new, the joy was: my app runs anywhere. Then someone said "great, now run 30 of them across 5 servers, restart the dead ones, and reroute traffic when one fails." That's the gap K8s fills.

THE PROBLEM

Without Kubernetes

  • Container crashed at 3am? You wake up
  • Traffic spiked? Manual scale-up, manually scale down later
  • Deploying v2? SSH into 10 servers in order
  • Server died? Pages, panic, partial outage
  • Need rollback? Hope you kept the last image around
Every line is a real on-call story. They're operational gaps that containers alone don't close. Each one is fixable manually — until you have a hundred of them at once.

THE PROBLEM

The 3 a.m. pager

Most outages are routine: something fell over, restart it. The cost is human attention at the worst hour.

90% of pages are "did you try restarting it." K8s does that automatically — your watch goes silent for the routine stuff, you only get woken for the hard problems.

THE INSIGHT

Containers are the unit.
Kubernetes is the
operating system.

If a container is a process, Kubernetes is the OS for your fleet. It schedules, restarts, networks, scales, and monitors — for any number of containers across any number of hosts.

PART 2

When to reach for it

K8s isn't free. Before we sell it, the same honesty as last time — when does it actually pay off.

WHEN TO USE

Use Kubernetes when

  • You run more than ~5 services
  • Traffic varies enough to need autoscaling
  • You need rolling, zero-downtime deploys
  • Self-healing is worth a platform investment
  • You're already on a cloud with managed K8s
Any one of these justifies it. Two or more and you're paying the operational tax without K8s anyway — better to spend it once on the platform.

WHEN NOT TO USE

Skip Kubernetes when

  • One service, one machine, low traffic
  • Static site behind a CDN
  • No platform team and no managed offering
  • Prototype with a 3-month runway
If you're a 2-person startup with one service, K8s is over-engineering. A single VM and `docker compose` will get you to product-market fit faster.

THE TRADEOFF

K8s isn't free

You trade manual ops for a learning curve and a control plane to operate.

There's a real complexity tax. The reason it's worth it: that complexity is centralized, debugged once, and pays off across every service you'll ever ship.

PART 3

From VMs to Kubernetes

A short story: how we got here, and why moving from VMs to K8s isn't just "smaller boxes."

THE JOURNEY

Four eras of deployment

2000s

Bare metal

One server, one app. Provisioning weeks.

2010s

VMs

Many guests per host. Provisioning hours.

Late 2010s

Containers

Light, fast, dense. Provisioning seconds.

Now

Kubernetes

Self-managing fleet. Declarative ops.

Each era reduced two costs: time-to-deploy and people-needed-to-deploy. K8s reduced a third: cognitive load — you describe what you want, not how to do it.

THE SHIFT

Stop managing servers.
Start describing
desired state.

This is the philosophical shift. With VMs you manage machines. With K8s you write a YAML that says "I want 3 of these," and the system makes it true and keeps it true.

VMs vs K8s

What changes when you move

VM era K8s era
Provision the VM, install the app Push image, write Deployment
SSH to fix issues kubectl logs, kubectl describe
Manual scaling, manual failover Replicas + HPA + self-heal
Snowflake servers Cattle, not pets
Static IPs and load balancers Services and Ingress
Five rows that change everything. The cattle-not-pets phrase is old but accurate: in K8s, no individual node or pod matters — the system replaces them all the time.

THE PAYOFF

What you stop doing

  • Logging into servers to restart things
  • Hand-rolling load balancer config
  • Patching OS packages on each VM
  • Writing bespoke deployment scripts
  • Calling cloud APIs to add capacity
Five jobs that disappear. They become declarative — described in YAML, executed by the platform. The platform pays the operational cost once, not per-service.

DEPLOY ONE SERVICE

Three eras, three commands

Manual 
ssh prod-01
sudo apt install nginx
git pull
npm ci && npm run build
pm2 start ecosystem.json
sudo systemctl reload nginx
# repeat on prod-02, prod-03 ...
Docker 
scp docker-compose.yml prod:/srv/
ssh prod
docker compose pull
docker compose up -d
# repeat on every host
Kubernetes 
git push origin main
# ArgoCD detects, pulls from SWR,
# applies manifest, rolling update,
# self-heals.
# Done.
Same goal — deploy one service to production. Manual: pages of steps, repeated per host. Docker: still per-host. K8s: one git push triggers everything, including rollout, healing, and observability.

DOCKER → KUBERNETES

Why we moved on from Docker alone

What Docker / compose can't do

  • Restart a dead container on another host
  • Roll out a new version without downtime
  • Scale by traffic, not by guessing
  • Reschedule when a host dies
  • Coordinate across more than one machine
  • Discover services by name across hosts

What Kubernetes does for you

  • Declarative spec → reconciled continuously
  • Rolling, blue-green, canary deploys
  • HPA, VPA, Cluster Autoscaler
  • Pod evictions and rescheduling
  • Service discovery built-in
  • Native ingress, secrets, configs, RBAC
This is the why. Docker shipped the package. Kubernetes ships the system that runs the packages. Every line on the right was a Friday-night incident on the left.

THE RACE

Three approaches.

PART 4

Kubernetes fundamentals

Just enough to be dangerous. The core objects you'll touch every day.

CLUSTER ANATOMY

Two halves of a cluster

Control plane

  • API server — the only door in
  • etcd — the source of truth
  • Scheduler — picks where pods land
  • Controllers — drive desired state

Worker nodes

  • kubelet — runs and reports pods
  • kube-proxy — networks them
  • container runtime — containerd, our case
  • your workloads — finally, the app
Control plane is the brain — it decides what should run. Workers do the running. In our setup, <Cloud Provider> manages the control plane, we just operate the workers.

THE UNIT

A Pod is the smallest
thing K8s schedules.

One or more tightly-coupled containers, sharing network and storage.

If a container is the unit of packaging, a Pod is the unit of scheduling. Most pods have one container. Multi-container pods exist for sidecars — logging agents, service meshes — but start with one.

FUNDAMENTALS

A Pod, in YAML

apiVersion: v1
kind: Pod
metadata:
  name: api
spec:
  containers:
    - name: api
      image: registry.example.com/yourorg/api:v1.4.2
      ports:
        - containerPort: 8080
      resources:
        requests: { cpu: 100m, memory: 128Mi }
        limits:   { cpu: 500m, memory: 256Mi }
You almost never write a raw Pod. You write Deployments and let them generate Pods. But this is the fundamental object — name, image, ports, and resource bounds.

FUNDAMENTALS

Deployment — desired state

"I want 3 of these, always. Roll updates without dropping traffic." 

apiVersion: apps/v1
kind: Deployment
metadata:
  name: api
spec:
  replicas: 3
  selector:
    matchLabels: { app: api }
  template:
    metadata:
      labels: { app: api }
    spec:
      containers:
        - name: api
          image: registry.example.com/yourorg/api:v1.4.2
Deployment manages a ReplicaSet which manages Pods. You only write the Deployment. K8s handles rollouts, rollbacks, and replacement.

FUNDAMENTALS

Service — a stable address

Pods come and go. A Service gives them one DNS name. 

apiVersion: v1
kind: Service
metadata:
  name: api
spec:
  selector: { app: api }
  ports:
    - port: 80
      targetPort: 8080
# In another pod:  curl http://api  → load-balanced across all api pods
A Service is internal load-balancing plus DNS. Other pods reach this service by name — they don't know or care which pod answers.

FUNDAMENTALS

Ingress — the front door

Routes external HTTPS traffic into Services. We use an ingress controller (e.g. Kong, Nginx, Traefik). 

apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: api
  annotations:
    nginx.ingress.kubernetes.io/rewrite-target: "/"
spec:
  ingressClassName: nginx
  rules:
    - host: api.example.com
      http:
        paths:
          - path: /
            pathType: Prefix
            backend:
              service: { name: api, port: { number: 80 } }
Service is internal. Ingress is the public edge — TLS, hostnames, paths. We run an ingress controller; it handles 28 ingresses across our cluster today.

FUNDAMENTALS

ConfigMaps & Secrets

  • ConfigMap — non-secret config (URLs, flags)
  • Secret — encoded values (passwords, tokens)
  • Mount as files OR inject as env vars
  • Don't bake either into the image
Image once, configure per environment. Same image runs in dev, staging, prod — only the ConfigMap and Secrets differ. We layer Vault on top of native Secrets for sensitive material.

FUNDAMENTALS

Namespaces — soft tenancy

Logical partitions for environments, teams, or apps.

PART 5

Flexibility & scaling

This is where K8s earns its place. Self-healing, autoscaling, and rolling deploys — for free, once you've described the desired state.

THE PROMISE

Describe the goal.
Kubernetes figures out
how to get there.

This is declarative ops. You don't say "start a pod" — you say "I want 3 pods running this image." K8s reconciles continuously: if one dies, it starts another. If you change the count, it adjusts.

SCALING

Three levels of autoscaling

HPA

Horizontal Pod Autoscaler

Adds or removes pod replicas based on CPU, memory, or custom metrics.

VPA

Vertical Pod Autoscaler

Adjusts CPU/memory requests for existing pods. Useful for right-sizing.

CA

Cluster Autoscaler

Adds or removes nodes (ECS instances) when pods can't be scheduled.

Three layers, top to bottom. HPA scales pods (most common). VPA right-sizes (less common). Cluster autoscaler adds nodes (cloud-managed). They compose: HPA tries to scale pods, can't fit, CA adds a node.

HPA

HPA, in YAML

apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: api
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: api
  minReplicas: 2
  maxReplicas: 10
  metrics:
    - type: Resource
      resource:
        name: cpu
        target:
          type: Utilization
          averageUtilization: 70

Already running on the GitOps controller, identity service, and ingress controller in our cluster.

This is a real config. CPU above 70% → scale up; below → scale down. Bound by min and max so a runaway loop can't bankrupt the company.

FLEXIBILITY

Rolling updates,
zero downtime

New pods come up healthy before old pods go down. 

kubectl set image deployment/api api=registry.example.com/yourorg/api:v1.4.3
# K8s spins up a v1.4.3 pod, waits for it to be Ready,
# then terminates a v1.4.2 pod. Repeat until done.

kubectl rollout status deployment/api
kubectl rollout undo   deployment/api  # rollback in seconds
Three commands: deploy, watch, rollback. The platform handles graceful shutdown, readiness probing, traffic shifting. You never write deployment scripts again.

PART 6

Our cloud · <Cloud>

Now the specific stack we run on. ECS for compute. CCE for orchestration. SWR Enterprise for images. All inside <Cloud Provider>.

THE STACK

<Cloud Provider> · three pieces

Compute

ECS

Elastic Cloud Server. The VM under each Kubernetes worker node.

Orchestration

CCE

Cloud Container Engine. Managed Kubernetes, control plane handled by <Cloud>.

Registry

SWR Enterprise

Private dedicated registry. Scanning, signing, replication, P2P pulls.

Three cloud services compose the platform. The compute service is just VMs. The managed Kubernetes service is K8s on top of those VMs, with the control plane managed for us. SWR Enterprise stores every image we run.

CCE SPECIFICS

What CCE gives us

  • Managed control plane — no etcd, no API server to babysit
  • Native ECS integration — node groups scale by adding ECS instances
  • Cloud storage classes — object store, shared file, SSD-backed, HDD-backed, SSD, plus standard CSI
  • Built-in SWR pulls — image pulls authenticate via cloud IAM
  • VPC-native networking — pods live on the VPC, not an overlay
  • One-click upgrades — minor version bumps without downtime
Six wins compared to running vanilla K8s on bare ECS. The big one for us: SWR + CCE auth happens natively, no kube secrets needed for image pulls. The storage variety matters when one workload needs SSD and another needs cheap object storage.
